# CoverAI - BDD Specification

Документ описывает целевое поведение MVP по исходной аналитике. Архитектурные
детали вынесены в отдельные документы; здесь фиксируются пользовательские и
системные сценарии, которые можно проверить через API, Telegram-бота, БД,
метрики и Docker Compose.

## Каналы доступа

CoverAI предоставляет два канала, каждый со своей ролью:

- **Telegram-бот** - основной потребительский канал. Минимальный путь от
  резюме до письма: /start, отправка резюме, ссылка на вакансию hh.ru.
  Конечные пользователи в основном работают через бот; email и пароль им не
  нужны.

- **REST API** - административный и интеграционный интерфейс с
  JWT-авторизацией. Закрывает три задачи:
  1. Администрирование системы: управление промокодами, аналитика, ручные
     корректировки балансов, рефанды, управление пользователями.
  2. Программный доступ для будущих внешних клиентов: веб-кабинета,
     партнерских интеграций, плагинов.
  3. Полная пользовательская функциональность для тех, кто сознательно хочет
     работать через API без бота.

Бот и API - адаптеры над общим сервисным слоем. Бизнес-правила (лимиты,
биллинг, тарифы) одинаковы во всех каналах. Аккаунты в каналах независимы;
объединение вне MVP.

## Глоссарий

- **Пользователь** - человек, который работает с CoverAI через REST API или
  Telegram.
- **Канал доступа** - способ взаимодействия с системой: Telegram-бот или
  REST API. Каналы работают через общий сервисный слой и применяют одинаковые
  бизнес-правила.
- **Аккаунт MVP** - запись пользователя привязана к одному каналу доступа:
  `telegram_id` для Telegram или `email` для REST API. Автоматическое
  объединение Telegram- и API-аккаунтов в MVP не выполняется.
- **Роль** - уровень доступа API-пользователя: `user` или `admin`.
- **Администратор** - пользователь с `role = admin`, имеющий доступ к
  административным эндпоинтам REST API. Создается seed-скриптом из переменных
  окружения при первом старте, не через `/auth/register`.
- **Профиль резюме** - один сохраненный текст резюме пользователя с названием.
- **Вакансия** - позиция на hh.ru, идентифицируемая ссылкой
  `https://hh.ru/vacancy/{id}`.
- **Запрос генерации** - запись `generation_requests` со статусом `pending`,
  `succeeded` или `failed`.
- **Письмо** - сгенерированное сопроводительное письмо.
- **Тон** - стиль письма: `formal`, `confident`, `concise`.
- **Кредит** - внутренняя единица оплаты успешной LLM-генерации.
- **Подписка** - платный доступ к тарифным возможностям Standard или Pro.
  Подписка управляет лимитами, историей и доступными тонами; кредиты остаются
  отдельным балансом для оплаты успешных генераций.
- **Подписочный период** - конкретный 30-дневный интервал платного тарифа,
  созданный успешной оплатой подписки. Продление до окончания текущего периода
  создает следующий период со `starts_at = current.expires_at`.
- **Лимит тарифа** - ограничение количества генераций за период.
- **Период тарифа** - интервал подсчета лимита: для Free календарный день в
  timezone `Europe/Moscow`, для Standard текущий подписочный период от
  `subscription.starts_at` до `subscription.expires_at`, для Pro лимита
  количества генераций нет.
- **Стартовые кредиты** - welcome bonus, который начисляется новому
  пользователю для первой генерации на Free.
- **Снимок генерации** - неизменяемая копия текста профиля, данных вакансии и
  tone, зафиксированная при постановке `generation_request` в очередь.

## Feature: REST API authentication and access control

> Как API-пользователь, я хочу получать JWT-токен и работать только со своими
> данными, чтобы система контролировала доступ к защищенным операциям.

### Scenario: AUTH-001 Регистрация внешнего клиента через API

```
Given  пользователь еще не зарегистрирован
  And  пользователь хочет программный доступ к системе
When   он отправляет POST /auth/register с email и password
Then   система создает пользователя с role = user, plan = free и email
  And  система начисляет стартовые кредиты для первой генерации
  And  система возвращает access_token
  And  access_token можно использовать как Bearer token
```

### Scenario: AUTH-002 Повторная регистрация с тем же email

```
Given  пользователь уже зарегистрирован с email user@example.com
When   он повторно отправляет POST /auth/register с тем же email
Then   API возвращает 409
  And  новый пользователь не создается
```

### Scenario: AUTH-003 Логин пользователя через API

```
Given  пользователь зарегистрирован
When   он отправляет POST /auth/login с корректными email и password
Then   система возвращает access_token
```

### Scenario: AUTH-004 Логин с неверным паролем

```
Given  пользователь зарегистрирован
When   он отправляет POST /auth/login с неверным password
Then   API возвращает 401
  And  access_token не выдается
```

### Scenario: AUTH-005 Защищенный endpoint без JWT

```
Given  пользователь не передал Bearer token
When   он запрашивает GET /users/me
Then   API возвращает 401
```

### Scenario: AUTH-006 Защищенный endpoint с истекшим JWT

```
Given  срок действия access_token истек
When   пользователь запрашивает защищенный endpoint с этим token
Then   API возвращает 401

When   пользователь повторно выполняет POST /auth/login
Then   система выдает новый access_token
```

### Scenario: AUTH-007 Защищенный endpoint с JWT

```
Given  пользователь получил access_token
When   он запрашивает GET /users/me с Authorization: Bearer {token}
Then   API возвращает данные текущего пользователя
```

### Scenario: AUTH-008 Обычный пользователь не может создать промокод

```
Given  пользователь авторизован через JWT с role = user
When   он отправляет POST /admin/promocodes
Then   API возвращает 403
```

### Scenario: AUTH-009 Администратор создает промокод

```
Given  пользователь авторизован через JWT с role = admin
When   он отправляет POST /admin/promocodes с валидными параметрами промокода
Then   API создает промокод
  And  API возвращает данные созданного промокода
```

### Scenario: AUTH-010 Swagger и OpenAPI доступны после запуска API

```
Given  API-сервис запущен
When   пользователь открывает /docs
Then   он видит Swagger UI

When   пользователь запрашивает /openapi.json
Then   схема содержит задокументированные API endpoints:
       - /health
       - /auth/register
       - /auth/login
       - /users/me
       - /profile
       - /generations
       - /generations/{id}
       - /generations/history
       - /billing/balance
       - /billing/transactions
       - /payments
       - /subscriptions
       - /subscriptions/current
       - /promocodes/redeem
       - /analytics/usage
       - /admin/users
       - /admin/users/{id}
       - /admin/users/{id}/balance-adjustment
       - /admin/users/{id}/subscription/expire
       - /admin/promocodes
       - /admin/analytics/overview
```

## Feature: Онбординг в Telegram

> Как новый Telegram-пользователь, я хочу быстро понять основной сценарий,
> чтобы сохранить резюме и отправить ссылку на вакансию.

```
Given  пользователь впервые открывает Telegram-бота
When   он отправляет /start
Then   бот создает или находит пользователя по telegram_id
  And  если пользователь создан впервые, система начисляет стартовые кредиты
  And  бот отправляет приветственное сообщение
  And  бот показывает основное меню
```

### Scenario: TG-ONBOARD-002 Повторный /start

```
Given  пользователь уже зарегистрирован в Telegram-боте
When   он отправляет /start
Then   бот не создает дубликат пользователя
  And  бот показывает приветственное сообщение и основное меню
```

## Feature: Каналы доступа и учетные записи

> Как владелец продукта, я хочу явно зафиксировать роли каналов и правила
> учетных записей, чтобы реализация и проверка соответствовали продуктовому
> позиционированию.

### Scenario: ACCOUNT-001 Конечный пользователь регистрируется через Telegram-бота

```
Given  человек впервые открывает Telegram-бота
When   он отправляет /start
Then   система создает пользователя с role = user, plan = free и telegram_id
  And  email и password_hash остаются пустыми
  And  пользователь получает доступ ко всему пользовательскому функционалу
       через бот без email-логина
```

### Scenario: ACCOUNT-002 Администратор работает через REST API по JWT

```
Given  администратор создан seed-скриптом из ADMIN_EMAIL и ADMIN_PASSWORD
When   он отправляет POST /auth/login с email и password
Then   API возвращает access_token с claim role = admin
  And  администратор получает доступ к /admin/* эндпоинтам
```

### Scenario: ACCOUNT-003 Telegram- и API-аккаунты независимы в MVP

```
Given  пользователь уже создан в Telegram-боте с telegram_id
When   тот же человек регистрируется через REST API с email и password
Then   система создает отдельного API-пользователя
  And  баланс кредитов Telegram-пользователя не переносится в API-аккаунт
  And  профиль, история и лимиты считаются отдельно для каждого аккаунта
  And  объединение аккаунтов остается вне MVP
```

## Feature: Управление профилем резюме

> Как пользователь, я хочу сохранить текст резюме один раз, чтобы использовать
> его для всех генераций.
>
> В MVP у пользователя есть ровно один профиль резюме. Профиль хранит название и
> сырой текст резюме; структурированный парсинг навыков, опыта и образования не
> выполняется.

### Scenario: PROFILE-001 Создание профиля текстом

```
Given  у пользователя нет профиля
When   пользователь отправляет текст резюме длиной от 100 символов
Then   бот сохраняет ResumeProfile(title, resume_text)
  And  бот сообщает, что профиль сохранен
  And  бот просит отправить ссылку на вакансию hh.ru
```

### Scenario: PROFILE-002 Создание профиля файлом

```
Given  у пользователя нет профиля
When   пользователь отправляет файл .txt, .md, .docx или .pdf с текстовым слоем
Then   бот извлекает текст из файла
  And  бот использует имя файла без расширения как title профиля
  And  если имя файла пустое, бот использует title = "Resume"
  And  бот применяет к извлеченному тексту те же ограничения длины, что и к текстовому вводу
  And  бот сохраняет ResumeProfile(title, resume_text)
  And  бот сообщает, что профиль сохранен
```

### Scenario: PROFILE-003 PDF без текстового слоя

```
Given  пользователь отправил PDF-файл
When   из PDF не удалось извлечь текст
Then   бот сообщает, что файл не удалось прочитать
  And  бот просит отправить другой файл или текст резюме
```

### Scenario: PROFILE-004 Неподдерживаемый формат файла

```
Given  пользователь отправил файл неподдерживаемого формата
When   бот пытается обработать файл
Then   бот сообщает, что поддерживаются .txt, .md, .docx и PDF с текстовым слоем
```

### Scenario: PROFILE-005 Текст резюме слишком короткий

```
Given  пользователь отправил текст резюме
When   текст короче 100 символов
Then   бот не сохраняет профиль
  And  бот сообщает, что текст резюме слишком короткий
```

### Scenario: PROFILE-006 Текст резюме слишком длинный

```
Given  пользователь отправил текст резюме длиннее 6000 символов
When   бот сохраняет профиль
Then   бот сохраняет первые 6000 символов
  And  бот сообщает, что текст был обрезан
```

### Scenario: PROFILE-007 Просмотр текущего профиля

```
Given  у пользователя есть профиль
When   он отправляет /profile
Then   бот показывает сохраненный текст профиля
```

### Scenario: PROFILE-008 Обновление единственного профиля

```
Given  у пользователя уже есть профиль
When   пользователь отправляет новый текст резюме или новый поддерживаемый файл
Then   бот обновляет существующий ResumeProfile
  And  бот сообщает, что профиль сохранен
```

### Scenario: PROFILE-009 API создает профиль резюме

```
Given  пользователь авторизован через JWT
  And  у пользователя нет профиля
When   он отправляет PUT /profile с title и resume_text длиной от 100 символов
Then   API создает ResumeProfile(title, resume_text)
  And  API возвращает созданный профиль
```

### Scenario: PROFILE-010 API обновляет единственный профиль резюме

```
Given  пользователь авторизован через JWT
  And  у пользователя уже есть профиль
When   он отправляет PUT /profile с новым resume_text
Then   API обновляет существующий ResumeProfile
  And  API не создает второй профиль
  And  API возвращает обновленный профиль
```

### Scenario: PROFILE-011 API получает текущий профиль

```
Given  пользователь авторизован через JWT
  And  у пользователя есть профиль
When   он отправляет GET /profile
Then   API возвращает title и resume_text текущего профиля
```

### Scenario: PROFILE-012 API не находит профиль

```
Given  пользователь авторизован через JWT
  And  у пользователя нет профиля
When   он отправляет GET /profile
Then   API возвращает 404
```

## Feature: Генерация сопроводительного письма через Telegram

> Как Telegram-пользователь, я хочу отправить ссылку на hh.ru и получить
> сопроводительное письмо без блокировки бота.

### Scenario: TG-GEN-001 Генерация письма на Free

```
Given  пользователь на плане Free
  And  у пользователя есть профиль
  And  у пользователя достаточно кредитов
  And  у пользователя не исчерпан дневной лимит тарифа
When   пользователь отправляет ссылку https://hh.ru/vacancy/123456
Then   бот ставит задачу generate_cover_letter в очередь
  And  бот сообщает, что вакансия принята
  And  тон генерации равен formal

When   worker успешно обрабатывает задачу
Then   worker сохраняет письмо
  And  worker списывает кредиты за успешную генерацию
  And  бот отправляет пользователю сгенерированное письмо
```

### Scenario: TG-GEN-002 Генерация письма на Standard или Pro

```
Given  пользователь на плане Standard или Pro
  And  у пользователя есть профиль
  And  у пользователя достаточно кредитов
  And  у пользователя не исчерпан лимит тарифа
When   пользователь отправляет ссылку https://hh.ru/vacancy/123456
Then   бот предлагает выбрать тон письма:
       - formal
       - confident
       - concise

When   пользователь выбирает тон
Then   бот ставит задачу generate_cover_letter в очередь с выбранным tone
  And  бот сообщает, что вакансия принята
```

### Scenario: TG-GEN-003 Генерация без профиля

```
Given  у пользователя нет профиля
When   пользователь отправляет ссылку на вакансию
Then   бот не ставит задачу в очередь
  And  бот просит сначала отправить резюме
```

### Scenario: TG-GEN-004 Недостаточно кредитов

```
Given  у пользователя недостаточно кредитов для генерации
When   пользователь отправляет ссылку на вакансию
Then   бот не ставит задачу в очередь
  And  бот сообщает, что недостаточно кредитов
```

### Scenario: TG-GEN-005 Лимит тарифа исчерпан

```
Given  у пользователя достаточно кредитов
  And  лимит тарифа пользователя исчерпан
When   пользователь отправляет ссылку на вакансию
Then   бот не ставит задачу в очередь
  And  бот сообщает, что лимит тарифа исчерпан
```

## Feature: Управление готовым письмом

> Как пользователь, я хочу быстро получить новый вариант письма для той же
> вакансии, чтобы выбрать лучший текст без повторного ввода ссылки.

### Scenario: LETTER-001 Перегенерация письма

```
Given  пользователь получил сгенерированное письмо
  And  у пользователя достаточно кредитов
  And  у пользователя не исчерпан лимит тарифа
When   пользователь нажимает Перегенерировать
Then   система ставит новую задачу генерации с тем же profile, vacancy и tone
  And  новая генерация считается отдельным generation_request

When   worker успешно завершает новую генерацию
Then   система сохраняет новое письмо как отдельный cover_letter
  And  система списывает кредиты как за обычную успешную генерацию
```

### Scenario: LETTER-002 Перегенерация без достаточного баланса

```
Given  пользователь получил сгенерированное письмо
  And  у пользователя недостаточно кредитов
When   пользователь нажимает Перегенерировать
Then   система не ставит новую задачу в очередь
  And  система сообщает, что недостаточно кредитов
```

### Scenario: LETTER-003 Перегенерация при исчерпанном лимите тарифа

```
Given  пользователь получил сгенерированное письмо
  And  у пользователя достаточно кредитов
  And  лимит тарифа пользователя исчерпан
When   пользователь нажимает Перегенерировать
Then   система не ставит новую задачу в очередь
  And  система сообщает, что лимит тарифа исчерпан
```

## Feature: Генерация сопроводительного письма через REST API

> Как API-пользователь, я хочу поставить генерацию в очередь и получить быстрый
> ответ API, чтобы LLM-запрос не блокировал HTTP request.

### Scenario: API-GEN-001 API ставит генерацию в очередь

```
Given  пользователь авторизован через JWT
  And  у пользователя есть профиль
  And  у пользователя достаточно кредитов
  And  у пользователя не исчерпан лимит тарифа
When   он отправляет POST /generations с vacancy_url и tone
Then   API возвращает 202 Accepted
  And  API возвращает queued = true
  And  API возвращает generation_request_id
  And  API ставит задачу generate_cover_letter в очередь
```

### Scenario: API-GEN-002 Free-пользователь не может выбрать нестандартный tone через API

```
Given  пользователь авторизован через JWT
  And  пользователь на плане Free
  And  у пользователя есть профиль
  And  у пользователя достаточно кредитов
  And  у пользователя не исчерпан дневной лимит тарифа
When   он отправляет POST /generations с tone = confident или tone = concise
Then   API возвращает 403
  And  задача не ставится в очередь
```

### Scenario: API-GEN-003 Free-пользователь генерирует через API без tone

```
Given  пользователь авторизован через JWT
  And  пользователь на плане Free
  And  у пользователя есть профиль
  And  у пользователя достаточно кредитов
  And  у пользователя не исчерпан дневной лимит тарифа
When   он отправляет POST /generations без tone

Given  пользователь на плане Pro

## Feature: Управление подпиской

> Как пользователь, я хочу перейти с Free на Standard или Pro, продлевать
> подписку и понимать, что произойдет с доступом при окончании оплаченного
> периода.

### Scenario: SUB-001 Free-пользователь покупает Standard или Pro

```
Given  пользователь на плане Free
  And  пользователь авторизован
When   он отправляет POST /subscriptions с plan = standard или plan = pro
Then   система создает subscription_payment_intent со статусом pending
  And  система передает платежному шлюзу сумму подписки, валюту, user_id, plan и payment_intent_id
  And  система возвращает пользователю данные для оплаты через платежный шлюз
  And  plan пользователя не меняется до подтверждения оплаты
```

### Scenario: SUB-002 Telegram-пользователь покупает Standard или Pro

```
Given  пользователь на плане Free работает через Telegram-бота
When   он выбирает апгрейд до Standard или Pro в меню тарифа
Then   бот создает subscription_payment_intent
  And  бот отправляет пользователю ссылку или кнопку оплаты платежного шлюза
  And  plan пользователя не меняется до подтверждения оплаты
```

### Scenario: SUB-003 Успешная оплата подписки активирует тариф

```
Given  существует subscription_payment_intent со статусом pending
When   платежный шлюз подтверждает успешную оплату подписки
Then   система создает Subscription со статусом active
  And  starts_at равен времени подтверждения оплаты
  And  expires_at равен starts_at + 30 дней
  And  система обновляет user.plan на оплаченный plan
  And  баланс кредитов пользователя не меняется
```

### Scenario: SUB-004 Продление активной подписки

```
Given  у пользователя есть активная подписка Standard или Pro
When   пользователь успешно оплачивает продление того же plan до expires_at
Then   система создает следующий подписочный период на 30 дней от текущего expires_at
  And  новый период лимитов начнется со starts_at следующего периода
  And  неиспользованные генерации предыдущего периода не переносятся
```

### Scenario: SUB-005 Смена тарифа во время активной подписки

```
Given  у пользователя есть активная подписка Standard
When   пользователь успешно оплачивает подписку Pro
Then   система завершает текущую Standard-подписку
  And  система сразу активирует Pro на 30 дней от времени подтверждения оплаты
  And  user.plan становится pro
```

### Scenario: SUB-006 Истечение подписки откатывает пользователя на Free

```
Given  у пользователя есть активная подписка Standard или Pro
  And  expires_at наступил
When   worker выполняет периодическую задачу обработки истекших подписок
Then   система переводит подписку в статус expired
  And  система обновляет user.plan на free
  And  доступ к истории Standard/Pro закрывается
  And  следующий лимит генераций считается как Free с текущего календарного дня
  And  баланс кредитов пользователя сохраняется
```

### Scenario: SUB-007 Отмена подписки пользователем вне MVP

```
Given  у пользователя есть активная подписка Standard или Pro
When   пользователь хочет отменить подписку до expires_at
Then   система сообщает, что отмена подписки пользователем вне MVP
  And  подписка продолжает действовать до expires_at
  And  автоматический возврат средств за неиспользованный период не выполняется

### Scenario: SUB-008 Даунгрейд Pro до Standard вне MVP

```
Given  у пользователя есть активная подписка Pro
When   пользователь хочет перейти на Standard до expires_at
Then   система сообщает, что даунгрейд вне MVP
  And  Pro продолжает действовать до expires_at
  And  пользователь может купить Standard после истечения Pro
```

### Scenario: SUB-009 Оплата подписки не заменяет баланс кредитов

```
Given  пользователь успешно оплатил подписку Standard или Pro
  And  balance_credits пользователя меньше стоимости генерации
When   пользователь запрашивает генерацию

```

### Scenario: SUB-010 Просмотр текущей подписки

```
Given  пользователь авторизован
When   пользователь отправляет GET /subscriptions/current
Then   API возвращает текущий plan пользователя
  And  если есть активная подписка, API возвращает starts_at и expires_at
  And  если активной подписки нет, API возвращает plan = free без active subscription

```

When   LLM вернул пустой текст

  And  кредиты не списываются

### Scenario: EXTERR-002 LLM недоступен

```

  And  если повторные попытки не помогли, worker вызывает fallback LLM model
  And  если fallback model тоже завершилась ошибкой, generation_request получает статус failed
  And  кредиты не списываются
```

When   hh.ru возвращает ошибку

  And  если повторные попытки не помогли, generation_request получает статус failed
  And  кредиты не списываются
